# DocDiff Roadmap — от MVP до релиза

> **Текущий статус:** MVP (ядро работает, тесты проходят, HTML-отчёт генерируется).  
> **Цель:** Выпустить v1.0 как стабильный CLI-инструмент для юристов, редакторов и аналитиков.

---

## Фаза 1. Полировка ядра (1–2 недели)

**Цель:** Исправить всё, что ломается на реальных документах.

- [ ] **Парсинг реальных документов**
  - Обработка таблиц в DOCX (не падать, извлекать как текст)
  - Поддержка колонтитулов / сносок (игнорировать или извлекать)
  - Работа с документами без стилей Heading (fallback на эвристики: размер шрифта, жирность)
  - Корректная работа с кириллицей и юникодом (проверить на реальных договорах)

- [ ] **Edge-cases в матчинге**
  - Пустые блоки (разделы без текста) — не падать
  - Блоки с одинаковым текстом, но разными заголовками — корректно определять "moved"
  - Документы с сильно изменённой структурой (более 50% блоков добавлены/удалены) — не деградировать в производительности

- [ ] **Багфиксинг и стабилизация**
  - Регрессионные тесты на 3–5 реальных документах (договоры, спецификации)
  - Проверка на утечки памяти при больших документах (50+ страниц)
  - Graceful degradation при отсутствии GPU (CPU-only fallback)

---

## Фаза 2. UX и интерфейсы (2–3 недели)

**Цель:** Продукт должен быть удобен не-программистам.

- [ ] **HTML-отчёт v2.0**
  - Тёмная тема (toggle)
  - Фильтрация по типу изменений (modified / added / removed / moved)
  - Навигация по якорям — оглавление слева
  - Показать "контекст" (окружающие абзацы) внутри изменений
  - Сохранение исходного форматирования текста (курсив, списки, таблицы в HTML)

- [ ] **Drag-and-drop веб-приложение (минимальное)**
  - Одностраничный HTML + FastAPI
  - Два поля для загрузки файлов
  - Кнопка "Сравнить" → результат в браузере
  - Не требует установки Python (или упаковка в exe)

- [ ] **Упаковка в standalone**
  - `pyinstaller` → `docdiff.exe` для Windows (пользователь не устанавливает Python)
  - Дистрибутив с встроенной моделью (без скачивания при первом запуске)
  - Или `pip install docdiff` из PyPI

---

## Фаза 3. Производительность и масштабируемость (1–2 недели)

**Цель:** 50-страничный документ обрабатывается < 10 секунд на CPU.

- [ ] **Оптимизация эмбеддингов**
  - Кэширование модели между запусками (уже частично есть через sentence-transformers)
  - Batch encoding с оптимальным размером батча
  - Опционально: более лёгкая модель (sentence-transformers/all-MiniLM-L6-v2 вместо L12-v2) для быстрого режима

- [ ] **Параллельная обработка**
  - Мультипроцессинг для эмбеддинга блоков
  - Асинхронный diff для независимых блоков

- [ ] **Прогресс-бар и обратная связь**
  - Для CLI: `tqdm` — показать этапы (парсинг, эмбеддинг, матчинг, diff)
  - Для веб: WebSocket-прогресс или Server-Sent Events

## Фаза 4. Дистрибуция и DevOps (1 неделя)

**Цель:** Продукт можно установить одной командой.

- [ ] **PyPI**
  - Собрать `wheel` и `sdist`
  - Опубликовать `pip install docdiff`
  - README с бейджами и quickstart

- [ ] **GitHub Releases**
  - CI/CD pipeline (GitHub Actions): тесты → сборка → релиз
  - Скомпилированный `.exe` для Windows в каждом релизе

- [ ] **Docker-образ (опционально)**
  - `docker run -v $(pwd):/docs docdiff old.docx new.docx`
  - Для корпоративных CI/CD пайплайнов

## Фаза 5. Документация и маркетинг (1–2 недели)

- [ ] **Документация**
  - Полное README с GIF-скриншотами (HTML-отчёт в действии)
  - Документация по API (докстринги → Sphinx → ReadTheDocs или GitHub Pages)
  - Руководство по использованию для юристов (не-программистов)
  - Частые вопросы (FAQ): "Почему этот блок не сопоставился?", "Как понизить порог?"

- [ ] **Примеры и витрина**
  - Папка `examples/` с реальными (анонимизированными) договорами
  - Сравнение с `diff` и `wdiff` (показать преимущество семантического подхода)
  - Бенчмарк: насколько быстрее/точнее человека на задаче "найти изменения в 50-страничном договоре"

- [ ] **Обратная связь и бета-тестирование**
  - Отдать 3–5 юристам / аналитикам на тестирование
  - Собрать фидбек и закрыть критические issue
